# llm.txt — rewriter_sft_svc

[project]
name = "prompt_optimisation"
description = "A modular prompt optimization project for clinical text, where a supervised fine-tuning microservice trains a Phi-3 Mini model to restructure messy clinical notes into strict structured clinical notes."

---------------------------------------------------------------------

[microservice]
service_name = "rewriter_sft_svc"
service_type = "training_microservice"
purpose = "Train a supervised fine-tuned Phi-3 Mini model on a synthetic dataset of paired clinical note and structured clinical note samples, and save the resulting SFT checkpoint for future inference and downstream RL-based prompt optimization."

This microservice is responsible only for supervised fine-tuning.
It takes a synthetically generated dataset as input and outputs a trained Phi-3 Mini SLM checkpoint into the sft_checkpoints/ directory.

The microservice must focus only on restructuring the clinical note into the required structured format.
It must not add, infer, assume, hallucinate, explain, summarize, or generate any unsupported content.

---------------------------------------------------------------------

[primary_goal]
Train a Phi-3 Mini model to learn the transformation:

original messy clinical note
→
structured clinical note

The trained model should act as a restructuring model only.

It should:
1. restructure the clinical note into the required fixed format
2. preserve only explicitly stated information
3. avoid adding any new content
4. avoid inference or hallucination
5. produce only the structured clinical note as output

It should not:
1. output the original clinical note
2. output the system instruction
3. output the extraction/output rules
4. output explanations or commentary
5. solve downstream tasks like ICD coding

---------------------------------------------------------------------

[input_dataset]
training dataset format:

{
    "clinical note": "<original clinical note>",
    "structured clinical note": "<structured clinical note>"
}

Example:

{
    "clinical note": "65 y/o M, h/o HTN, DM2, c/o chest pain x2 days, sweating, nausea, ECG done, trop pending, ?ACS, no fever",
    "structured clinical note": "Patient Demographics:\nAge:\n65\nGender:\nMale\nPrimary Diagnosis:\nNot specified\nSecondary Diagnoses:\nHypertension\nType 2 Diabetes Mellitus\nSymptoms:\nChest pain\nSweating\nNausea\nDuration:\n2 days\nInvestigations:\nECG\nTroponin pending\nProcedures:\nNot specified\nComorbidities:\nHypertension\nType 2 Diabetes Mellitus\nRisk Factors:\nNot specified\nComplications:\nNot specified"
}

The dataset contains no instruction field.
Therefore, the training pipeline must automatically prepend a fixed system instruction during preprocessing.

---------------------------------------------------------------------

[system_instruction]
The following system instruction must be added internally to every training sample:

"You are a clinical information restructuring assistant. Convert the given messy clinical note into the exact structured clinical note format using only explicitly stated medical information. Do not add, infer, assume, hallucinate, explain, paraphrase, summarize, or generate any extra content outside the required structured output."

This system instruction is used only internally for training and inference formatting.
It must not appear in the final generated output.

---------------------------------------------------------------------

[base_model]
model_name = "microsoft/Phi-3-mini-4k-instruct"
training_method = "Supervised Fine-Tuning (SFT)"
recommended_strategy = "QLoRA"
alternative_strategy = "LoRA"
not_recommended_by_default = "Full fine-tuning unless high compute resources are available"

Reason for using Phi-3 Mini:
1. lightweight and efficient
2. suitable for instruction-following
3. practical for domain-specific supervised fine-tuning
4. easy to save and reload for later inference
5. useful as a strong initialization point for future RL fine-tuning

[restructuring_scope]
This service must only focus on restructuring the clinical note.

Important constraints:
1. Only restructure content already present in the original clinical note.
2. Do not add any new medical fact.
3. Do not infer any unstated symptom, diagnosis, procedure, or comorbidity.
4. Do not hallucinate.
5. Do not explain the note.
6. Do not summarize the note.
7. Do not return narrative prose.
8. Do not output anything except the structured clinical note.

The structured clinical note should contain only the newly structured clinical note.
It must not include:
1. output rules
2. original clinical note
3. system prompt
4. instruction text
5. commentary
6. headings such as "Clinical Note" or "Structured Format"

---------------------------------------------------------------------

[required_output_format]
The model output must always strictly follow this exact structure:

Patient Demographics:
Age:
Gender:
Primary Diagnosis:
Secondary Diagnoses:
Symptoms:
Duration:
Investigations:
Procedures:
Comorbidities:
Risk Factors:
Complications:

All fields must appear in the exact order shown above.

---------------------------------------------------------------------

[output_rules]
RULES:
1. Extract only information explicitly mentioned in the clinical note.
2. Do NOT infer, assume or hallucinate medical information.
3. Do NOT add clinical explanations, reasoning, or commentary.
4. Do NOT paraphrase, rewrite, or summarize the clinical note. Only extract and organize relevant medical information.
5. Include only medically relevant information useful for diagnosis and clinical coding.
6. Do NOT add any extra text outside the required output format.
7. Do NOT include headings such as "Clinical Note", "Structured Format", or similar labels.
8. Follow the exact output structure provided. Do not add or remove fields.
9. If information for a field is not present in the clinical note, write: Not specified.
10. Multiple items within the same field must appear on separate lines.
11. Use clear medical terminology when listing diagnoses, symptoms, investigations, or procedures.
12. Do not repeat the same information across multiple fields.
13. Do not include explanations, justifications, or interpretation of the note.
14. Ensure the output contains ONLY the structured fields defined in the output format.
15. All fields must appear in the output exactly in the specified order, even if the value is "Not specified".
16. Extract concise medical terms or phrases instead of copying long narrative sentences from the note.
17. Symptoms must not be listed as diagnoses unless explicitly documented as a diagnosis in the note.

These rules define how the output should be generated.
They are part of training and evaluation logic only.
They must not be included in the model output.

---------------------------------------------------------------------

[training_pipeline]
Step 1:
Load the synthetic training dataset.

Step 2:
Validate dataset schema:
- clinical note
- structured clinical note

Step 3:
Attach the fixed internal system instruction to every sample.

Step 4:
Convert each sample into Phi-3 Mini chat-style training format.

Step 5:
Tokenize the dataset.

Step 6:
Load the Phi-3 Mini base model.

Step 7:
Apply QLoRA adapters.

Step 8:
Train using supervised fine-tuning.

Step 9:
Evaluate on validation and test sets.

Step 10:
Save the trained checkpoint to sft_checkpoints/.

Step 11:
Generate testing_summary.md inside the service directory.

---------------------------------------------------------------------

[training_format]
Each raw dataset sample:

{
    "clinical note": "<original clinical note>",
    "structured clinical note": "<structured clinical note>"
}

must be converted internally into:

{
  "messages": [
    {
      "role": "system",
      "content": "You are a clinical information restructuring assistant. Convert the given messy clinical note into the exact structured clinical note format using only explicitly stated medical information. Do not add, infer, assume, hallucinate, explain, paraphrase, summarize, or generate any extra content outside the required structured output."
    },
    {
      "role": "user",
      "content": "<original clinical note>"
    },
    {
      "role": "assistant",
      "content": "<structured clinical note>"
    }
  ]
}

The assistant content must contain only the structured clinical note.
It must not contain rules, prompt text, or original note text.

[connection_to_rl]
This SFT model serves as the initial policy for later RL fine-tuning.

Why:
1. SFT teaches the base restructuring behavior
2. RL can later improve output quality further
3. RL can optimize against downstream rewards such as format compliance, hallucination reduction, extraction precision, and downstream task usefulness

The current microservice only handles the SFT stage and checkpoint generation.

The rewriter_sft_svc microservice takes a synthetic paired dataset in the format:

{
    "clinical note": "<original clinical note>",
